# 仪表盘重构设计

日期：2026-06-07

项目：`sim_delivery_system`

目标：在不回退现有正式系统骨架的前提下，将首页仪表盘重构为“保留原型气质、重新做正式版”的版本，使视觉风格、图表类型和数据组织更接近 `d:\0-wiki\prototypes\sim-project-dashboard.html`，同时继续使用当前正式版的路由和后端接口。

## 1. 范围

本次只重构首页仪表盘，不改动以下页面的信息架构：

- `新建项目`
- `项目列表`
- `项目详情`

本次重构关注三件事：

- 首页视觉风格向原型靠拢
- 首页图表类型向原型靠拢
- 首页接口数据结构向原型友好型结构靠拢

本次不做：

- 客户门户
- 多人权限
- 打包流程落地
- 目录自动创建
- 项目详情回到首页

## 2. 设计原则

### 2.1 页面职责不回退

首页继续承担“管理总览”职责，而不是重新变成“操作台”。

首页回答的问题仍然是：

- 当前项目盘子有多大
- 项目主要卡在哪些阶段
- 最近交付压力如何变化
- 哪些项目金额更值得重点关注
- 本周哪些风险需要优先处理

项目详情、编辑、交付动作仍保留在独立页面中。

### 2.2 风格靠近原型，结构服务正式版

正式版不要求一比一还原原型，但要保留原型中已经被验证有效的气质：

- 深色商务科技风
- 更强的卡片层级
- 更明显的留白和分区
- 更像展示型仪表盘而不是普通后台表格
- 图表有独立视觉表达，而不是全部退化为统一条形块

### 2.3 数据结构为图表服务

后端接口不再只提供“能凑出来”的字段，而是按图表需要输出更直接的数据结构，减少前端临时转换和硬编码。

## 3. 首页结构

首页保持当前正式版的整体壳层：

- 左侧导航
- 右侧主内容区

首页主内容重构为以下层级：

### 3.1 顶部标题区

保留正式版页面头部的系统语义，但样式向原型靠拢。

包含：

- 页面眉标
- 标题
- 一句说明文案
- 后端连接状态
- 主要操作按钮

视觉目标：

- 更像原型中的标题区
- 避免当前版本过于朴素的企业后台感

### 3.2 核心指标区

保留 4 个核心 KPI：

- `项目总数`
- `总项目金额`
- `临近交付`
- `可打包项目`

表现方式：

- 使用独立指标卡
- 保留更强的层次、圆角、背景渐变和辅助说明
- 允许保留正式系统里的业务文案，但表现风格向原型靠拢

### 3.3 图表区

图表区改为 3 个核心图 + 1 组摘要/提醒卡的组合：

- `项目阶段分布`
- `月度交付趋势`
- `金额 Top 项目`
- `经营摘要 / 风险提醒`

整体布局参考原型的双列信息组织，不恢复首页项目详情。

## 4. 图表设计

### 4.1 项目阶段分布

用途：回答“当前项目堆积在哪个阶段”。

实现方式：

- 使用横向条形分布图
- 风格接近原型中的 `stage-chart`
- 每一行显示：
  - 阶段名
  - 条形轨道
  - 数值

数据源：

- `stageDistribution: [{ label, value }]`

### 4.2 月度交付趋势

用途：回答“最近交付压力是否上升”。

实现方式：

- 使用 SVG 折线图 + 面积填充
- 保留原型中：
  - 网格线
  - 折线
  - 面积阴影
  - 数据点
  - 横轴月份标签

不再使用当前正式版里将趋势图退化为普通条形行的做法。

数据源：

- `deliveryTrend: [{ month, value }]`

### 4.3 金额 Top 项目

用途：回答“金额集中在哪些项目”。

实现方式：

- 使用接近原型的排行条
- 每一行显示：
  - 项目名
  - 排行进度条
  - 金额数值

数据源：

- `topAmounts: [{ label, value }]`

### 4.4 经营摘要 / 风险提醒

用途：减少首页纯图表化导致的信息解释成本。

实现方式：

- 经营摘要采用卡片列表
- 风险提醒保留当前正式版的“本周优先事项”语义
- 视觉样式向原型的摘要卡和提醒卡靠拢

数据源建议拆为两个字段：

- `summaryCards`
- `riskReminders`

如果后端暂时不拆，前端可从已有数据先组合出摘要，但最终目标是后端直接提供。

## 5. 数据接口设计

首页仪表盘接口继续保留：

- `GET /api/dashboard`

但返回结构调整为更接近原型语义：

```json
{
  "projectCount": 18,
  "totalAmountWan": 286,
  "nearDelivery": 4,
  "readyToPack": 3,
  "stageDistribution": [
    { "label": "立项", "value": 2 },
    { "label": "建模", "value": 4 }
  ],
  "deliveryTrend": [
    { "month": "1月", "value": 2 },
    { "month": "2月", "value": 3 }
  ],
  "topAmounts": [
    { "label": "项目A", "value": 26 }
  ],
  "summaryCards": [
    { "title": "金额集中度较高", "detail": "前两大项目占比较高", "level": "normal" }
  ],
  "riskReminders": [
    { "title": "优先推进临近交付项目", "detail": "某项目需在 2 天内完成说明确认", "level": "high" }
  ]
}
```

### 5.1 后端生成原则

后端应尽量基于真实项目数据生成：

- `projectCount`
- `totalAmountWan`
- `nearDelivery`
- `readyToPack`
- `stageDistribution`
- `topAmounts`

允许以下内容先保留为可控样例数据，再逐步演进：

- `deliveryTrend`
- `summaryCards`
- `riskReminders`

## 6. 前端实现原则

### 6.1 保留现有路由骨架

继续使用当前基于查询参数的路由：

- `?page=dashboard`
- `?page=create`
- `?page=projects`
- `?page=detail&id=...`

### 6.2 仪表盘作为独立呈现层重写

不在当前 `renderDashboard()` 上小修小补，而是按首页结构整体重写，避免保留当前“全部统一条形块”的中间态实现。

### 6.3 图表用原生 SVG / DOM 实现

本次不额外引入图表库。

原因：

- 原型已证明原生 SVG 足够表达需要的图表
- 能更稳定控制风格一致性
- 当前项目仍是轻量前端骨架，避免为了首页图表引入额外依赖

## 7. 与现有版本的差异

当前正式版的问题：

- `交付趋势` 被简化成普通条形行，不像原型
- `金额 Top 项目` 的视觉表达较弱
- 缺少原型里的经营摘要层次
- 首页图表区整体气质偏普通后台，而不是高质量仪表盘

重构后的目标状态：

- 首页仍是正式产品页
- 但图表观感明显回到原型认可的方向
- 数据来自正式接口而不是纯演示页

## 8. 后续开发顺序

仪表盘重构完成后，下一批功能按以下顺序推进：

1. `新建项目体验补强`
2. `项目列表可筛选可排序`
3. `项目详情的阶段更新`
4. `交付清单与打包入口`

## 9. 验收标准

满足以下条件视为本次完成：

- 首页视觉明显接近原型气质
- 首页图表类型恢复为：
  - 阶段分布横条图
  - 月度交付趋势折线/面积图
  - 金额排行图
- `GET /api/dashboard` 返回结构能够直接支撑这些图表
- 首页仍保持正式系统导航与多页面结构
- `项目详情` 不回到首页
- Windows 浏览器访问 `http://localhost:5173/` 可正常看到重构后的首页
