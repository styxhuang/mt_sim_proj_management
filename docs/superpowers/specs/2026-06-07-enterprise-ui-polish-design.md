# 企业控制台细节收敛设计

日期：2026-06-07

项目：`sim_delivery_system`

目标：在不改动现有信息架构和前后端接口边界的前提下，将当前产品从“可用的深色单文件演示页”收敛为“更稳定、更克制、更像企业内部控制台”的版本，重点优化全局视觉系统、页面布局节奏、字体层级、交互反馈和轻量全局设置。

## 1. 范围

本次设计覆盖以下页面与全局层：

- `项目仪表盘`
- `新建项目`
- `项目列表`
- `项目详情`
- 全局样式 token
- 全局交互反馈
- 轻量设置入口

本次不改变以下内容：

- 不新增新的业务页面
- 不重做路由结构
- 不引入新的前端框架
- 不改成多文件组件化架构
- 不新增复杂账户体系
- 不引入后端配置中心
- 不变更已有项目字段和核心接口语义

## 2. 设计目标

本次收敛要同时达到 4 个目标：

- 视觉上更像正式企业控制台，而不是偏展示型 demo
- 页面层级更稳定，用户能更快识别标题、状态、数据和操作
- 操作反馈更明确，减少“点了但不确定是否生效”的感觉
- 通过轻量设置补齐产品感，但不把本轮工作扩大成完整设置系统

## 3. 非目标

本次不追求以下方向：

- 不做品牌级视觉重塑
- 不做亮色主题和主题切换系统
- 不做移动端优先设计
- 不做大屏展示型改版
- 不做权限、协作、审批、通知中心
- 不做完整“系统设置页”

## 4. 总体策略

本次采用“中度收敛”策略，而不是轻微调色或页面重构。

这意味着：

- 保留当前 `侧边栏 + 主内容区` 的产品骨架
- 保留现有单文件实现方式
- 保留暗色商务科技风基底
- 收敛视觉噪声，减少过强的装饰感和漂浮感
- 统一 spacing、字号、状态、控件尺寸和反馈语义
- 通过有限结构重排提升正式感，但不推翻现有页面结构

## 5. 全局视觉系统

### 5.1 视觉基调

整体视觉从“深色玻璃感 demo”收成“深色企业控制台”。

保留：

- 暗色背景
- 轻微品牌蓝青色强调
- 卡片化布局
- 温和的层次过渡

弱化：

- 过多的泛光和装饰性高亮
- 大面积渐变造成的展示感
- 模块之间几乎同权的强对比
- hover 时过强的浮起感

### 5.2 颜色层级

全局颜色需要形成明确的层级，而不是只用少量变量覆盖所有场景。

建议收敛为以下层次：

- `背景层`：页面底色、全局氛围层
- `容器层`：主工作区与侧边栏底板
- `面板层`：卡片、表单、表格行、空态块
- `强调层`：主按钮、焦点、选中态、可操作入口
- `状态层`：成功、警告、风险、完成、待处理

主色只负责：

- 主按钮
- 焦点 ring
- 选中态
- 链接和可点击重点

状态色统一映射，不允许每个区块自行表达同一状态。

### 5.3 卡片体系

卡片分成 3 类：

- `主卡片`：仪表盘核心图表、详情主编辑区
- `次卡片`：摘要块、辅助信息区、风险提醒
- `静态信息块`：小型说明、空态、记录项

三类卡片共享同一 token 体系，但在以下方面拉开区别：

- 阴影强度
- 边框清晰度
- 内边距
- 标题强度
- 背景透明度

目的不是做出完全不同的皮肤，而是让用户能自然感知“哪块最重要”。

### 5.4 控件规格

所有可操作控件统一一套尺寸语言：

- 主输入框与下拉框使用统一高度
- 主按钮与次按钮高度分层
- 标签、胶囊、状态 pill 保持更紧凑尺寸
- 表单字段圆角、边框、内边距一致

重点是让页面看起来“整”，而不是每种控件都有一点不同。

### 5.5 动效原则

动效采用“弱动态、强反馈”原则：

- hover 轻微变化即可
- active 更偏压感而不是漂浮
- focus 明确可见
- loading 明确但不抢眼
- 成功反馈短暂、克制

动效服务于可预期性，而不是装饰。

## 6. 页面布局节奏

### 6.1 整体框架

保留当前：

- 左侧导航
- 右侧主内容区

但主内容区要从“卡片堆叠”收成“有工作流层次的页面”。

统一页面结构节奏：

1. 页面头部
2. 工具条或主要操作区
3. 主体内容区
4. 辅助说明或空态

### 6.2 页面头部

每页头部统一为两层结构：

- 第一层：标题、简述、关键状态
- 第二层：当前页操作或筛选工具

不再让标题、说明、筛选、按钮混在同一层里争夺注意力。

### 6.3 仪表盘

仪表盘从“展示型 hero”收成“业务总览头部 + 核心模块区”。

目标：

- 首页第一屏先回答业务判断问题
- KPI、图表、提醒之间拉开主次
- 摘要和提醒保留，但不抢占图表和数据的主视觉

允许保留现有图表结构，但页面节奏上要更紧凑、更企业化。

### 6.4 项目列表

项目列表页强调“工具条 + 列表主体”的工作台感。

布局策略：

- 筛选和排序独立成工具条
- 列表标题行更明确
- 行内信息按主次分组
- 项目名、客户、状态、金额、日期之间层级拉开

列表页不追求展示感，追求扫读效率。

### 6.5 项目详情

详情页改造成真正的单项目维护工作台，分成 4 个主要区块：

- 基础信息与核心状态编辑
- 阶段与进度信息
- 交付清单维护
- 打包记录维护

这 4 个区块必须体现明确优先级，不再像若干同权卡片并列。

### 6.6 新建项目

新建页强调“单任务录入”的稳定感。

设计原则：

- 弱化解释文字
- 强化字段顺序
- 突出提交动作
- 保持表单扫描顺序自然

### 6.7 间距系统

间距要形成稳定节奏：

- 页面级间距：大
- 模块级间距：中
- 字段组间距：中小
- 标签和说明间距：小

不要求写成完整数学系统，但在实现上必须收敛到少量固定值，避免页面各处间距随意。

## 7. 字体与信息层级

### 7.1 字体策略

当前页面的英文无衬线风格可以保留，但中文显示必须以系统中文字体链为优先，避免中文质感发飘。

目标效果：

- 中文更稳
- 数字更清晰
- 标题更像正式产品

### 7.2 字阶设计

至少建立以下层级：

- 页面主标题
- 区块标题
- 卡片标题
- 正文
- 辅助文案
- 标签与状态文字

这些层级必须在字号、字重、颜色、行高上形成稳定差异。

### 7.3 数据优先级

金额、进度、交付日期、项目数量等关键业务数据要比正文更“硬”。

实现方向：

- 数据值在字号或字重上单独处理
- 单位和说明弱化
- 让用户先看到业务判断点，再看到解释

### 7.4 表单信息层级

表单内至少区分以下 4 层：

- 字段标题
- 当前输入值
- 辅助说明
- 错误反馈

目的：用户扫一眼就知道“这是字段名、这是值、这是帮助、这是报错”。

### 7.5 列表阅读层级

列表内主信息和辅助信息必须分层：

- 主信息：项目名、核心状态
- 次信息：客户、模板、路径等辅助字段
- 关键数字：金额、进度、日期

列表设计优先支持快速判断，而不是完整阅读。

## 8. 交互反馈

### 8.1 按钮状态

按钮至少统一以下状态：

- 默认
- hover
- active
- disabled
- loading

主按钮、次按钮、弱按钮需要清晰分层，不允许仅靠颜色深浅微调。

### 8.2 表单反馈

当前仅用 `alert` 的方式不够企业化。

本次应补齐：

- 表单内联错误或状态提示
- 聚焦态增强
- 提交中态
- 保存成功后的短暂成功提示

错误反馈优先贴近表单和操作按钮，而不是全靠浏览器弹窗。

### 8.3 列表与详情交互

要让用户明确知道：

- 哪些区域可点进详情
- 哪些按钮会改变状态
- 哪些区域只是静态信息

hover、边框、背景、文案样式共同承担“是否可操作”的表达。

### 8.4 筛选与排序反馈

筛选器要有明确的“当前生效”感，包括：

- 选中态清晰
- 重置入口易发现
- 无结果状态文案自然
- 排序逻辑变化后界面反馈可感知

### 8.5 空态与缺省态

以下空态需要统一成正式产品文案：

- 没有匹配的项目
- 还没有登记打包记录
- 当前页数据加载失败
- 后端不可用时使用示例数据

空态不应像说明文档，也不应像原型占位符。

## 9. 轻量全局设置

### 9.1 目标

本轮只补“轻量设置入口”，不补完整设置系统。

设置的作用是增加产品完成度，而不是引入复杂业务。

### 9.2 设置范围

本轮适合放入前端本地偏好的设置包括：

- 默认首页
- 项目列表默认排序
- 页面密度偏好
- 是否显示辅助说明文案

### 9.3 存储方式

优先采用前端本地存储，不新增后端接口。

原因：

- 作用域只在本地使用体验
- 无账户体系，不需要跨设备同步
- 成本低，适合当前阶段

### 9.4 入口形式

本轮推荐采用“页面右上角入口 + 弹出式轻面板”的形式。

不采用完整独立设置页，也不新增侧边栏一级导航。

这样做的原因是：

- 入口易被发现
- 不打断主流程
- 实现成本低
- 适合承载少量本地偏好

## 10. 实施边界

为了控制本轮范围，实现上应遵守以下边界：

- 主要修改 `frontend/public/index.html`
- 后端只做设置所需接口时才扩展，但当前设计默认不需要新增后端接口
- 不把本轮演进成组件拆分重构
- 不为视觉打磨引入第三方 UI 库

## 11. 验收标准

本轮完成后应满足：

- 页面整体观感明显更像正式企业控制台
- 页面头部、工具条、内容区节奏统一
- 中文与数字显示更稳定，标题和数据层级更清晰
- 表单、按钮、筛选、列表交互反馈更明确
- 不再只依赖 `alert` 表达保存结果
- 至少存在一个可用的轻量全局设置入口
- 保持现有功能不回退
- 保持现有接口与页面路由可用

## 12. 推荐实现顺序

建议按以下顺序推进：

1. 先收敛全局 token：颜色、间距、字阶、圆角、阴影、控件高度
2. 再统一页面头部与工具条结构
3. 然后细化列表页和详情页的阅读与操作层级
4. 再补表单和操作反馈
5. 最后加入轻量全局设置与本地偏好持久化

这样能先解决“全局不统一”的问题，再解决局部页面细节问题，避免后改前推翻。
